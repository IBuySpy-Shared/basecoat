# Integrate and Audit BaseCoat

**How to invoke:**

```text
integrate and audit BaseCoat from https://github.com/YOUR-ORG/YOUR-REPO
```

Or, if you are already inside the target repo:

```text
integrate and audit BaseCoat
```

---

## What This Prompt Does

This prompt runs a two-phase guided workflow:

1. **Integrate** — inspects the target repo, generates the exact sync commands for the team's environment, configures `.basecoat.yml`, and walks through first-sync validation.
2. **Audit** — runs a baseline assessment of the repo's Copilot readiness against the BaseCoat standard: instruction coverage, agent relevance, skill gaps, naming consistency, and CI configuration.

It produces a prioritized report the team can act on immediately. No guessing, no docs-hunting.

---

## Workflow

### Phase 1: Discover

Examine the target repository before recommending anything. Collect:

1. **Tech stack** — languages, frameworks, primary build tools (infer from `package.json`, `*.csproj`, `go.mod`, `pyproject.toml`, `Makefile`, `Dockerfile`, etc.)
2. **Existing Copilot customizations** — check `.github/copilot-instructions.md`, `.github/agents/`, `.github/skills/`, `.github/instructions/`, `.github/prompts/`. Note what exists and what version, if any (`version.json`, `.basecoat.yml`).
3. **CI/CD setup** — check `.github/workflows/`. Note: node runners, environment, secret names, matrix targets, caching patterns.
4. **Team size signal** — look at `CODEOWNERS`, `CONTRIBUTING.md`, or branch protection rules. Solo project vs. team changes the recommended rollout path.
5. **Open issues and PRs** — briefly check if any Copilot-related issues or PRs are already open. Don't block on this; note and continue.

Summarize findings in a brief **Repo Profile** table before proceeding.

---

### Phase 2: Integrate

#### Step 1 — Choose a sync method

Based on the discovery, recommend one of three paths:

| Path | When to use |
|---|---|
| **Direct sync (quickstart)** | Solo or small team; public repo; trying BaseCoat for the first time |
| **Fork first** | Enterprise org; need to customize assets before rolling out; private fork preferred |
| **Org-wide rollout** | Multiple consumer repos; centralized governance; see `docs/guides/enterprise-rollout.md` |

Print the exact commands for the chosen path. Use the actual repo URL, org name, and detected OS/shell from context (PowerShell vs. Bash).

**Direct sync example:**

```bash
# From the root of the target repo — macOS/Linux
export BASECOAT_REPO='https://github.com/IBuySpy-Shared/basecoat.git'
curl -fsSL https://raw.githubusercontent.com/IBuySpy-Shared/basecoat/main/sync.sh | bash
```

```powershell
# Windows PowerShell
$env:BASECOAT_REPO = 'https://github.com/IBuySpy-Shared/basecoat.git'
irm https://raw.githubusercontent.com/IBuySpy-Shared/basecoat/main/sync.ps1 | iex
```

#### Step 2 — Generate `.basecoat.yml`

Draft a `.basecoat.yml` file tailored to the repo. At minimum include:

```yaml
# .basecoat.yml — BaseCoat consumer configuration
# Auto-generated by the integrate prompt. Commit this file.

source: https://github.com/IBuySpy-Shared/basecoat.git
ref: main    # pin to a tag (e.g. v3.25.0) for stability in production

# Agents to include (remove any that don't fit the stack)
agents:
  - solution-architect
  - code-review
  - sprint-planner
  - security-review
  # Add stack-specific agents discovered in Phase 1

# Skills to include
skills:
  - azure-diagnostics     # if Azure detected
  - database-migration    # if DB migrations detected

# Instructions applied globally
instructions:
  - governance
  - token-economics

# Instructions applied to specific file patterns (inferred from tech stack)
scoped_instructions: []   # populated by the AI based on detected languages

sync:
  exclude: []   # paths to skip (e.g. archive/)
```

Adjust `agents`, `skills`, and `instructions` based on the tech stack discovered in Phase 1. Explain each inclusion briefly.

#### Step 3 — Validate the sync

After the user runs sync, check for success by verifying:

- `.github/agents/` contains at least one `.agent.md` file
- `.github/copilot-instructions.md` was created or updated
- `version.json` is present in `.github/` with a `basecoat_version` field

If any check fails, diagnose the likely cause (missing `BASECOAT_REPO`, network, permissions) and provide the fix.

---

### Phase 3: Audit

Run a baseline audit. Do not assume the sync ran perfectly or that the repo started from zero. Check what is actually there.

#### Audit dimensions

#### A. Instruction coverage

| Check | Good | Needs work |
|---|---|---|
| Global instruction file exists and is ≤2KB | ✅ | ❌ File missing or >2KB (adds token cost to every session) |
| Scoped instructions exist for primary language | ✅ | ❌ All guidance is global; no per-pattern narrowing |
| No duplicate guidance between global and scoped files | ✅ | ❌ Same rule appears in multiple instruction files |
| Governance instructions present | ✅ | ❌ Missing |

#### B. Agent relevance

| Check | Good | Needs work |
|---|---|---|
| At least one agent matches primary tech stack | ✅ | ❌ No stack-specific agent |
| Agents have all three required sections: Inputs, Workflow, Output | ✅ | ❌ Missing section |
| Agents reference skills via `allowed_skills` where appropriate | ✅ | ❌ Skills exist but agents don't reference them |

#### C. Skill coverage

| Check | Good | Needs work |
|---|---|---|
| At least one skill per major domain in use | ✅ | ❌ Skills directory is empty |
| Skills have `name` and `description` frontmatter | ✅ | ❌ Missing frontmatter |

#### D. CI integration

| Check | Good | Needs work |
|---|---|---|
| Workflow calls `check-basecoat-version-callable.yml` or equivalent | ✅ | ❌ No drift detection |
| No BaseCoat workflow files reference undefined secrets | ✅ | ❌ Missing secret triggers silent failure |
| `run-tests.ps1` / equivalent invoked in CI | ✅ | ❌ No automated validation |

#### E. Naming and taxonomy

| Check | Good | Needs work |
|---|---|---|
| Agent files follow `kebab-case.agent.md` convention | ✅ | ❌ Misnamed files |
| Instruction files follow `kebab-case.instructions.md` | ✅ | ❌ Misnamed files |
| No deprecated BaseCoat terminology in docs | ✅ | ❌ "Base Coat" (space), "plugin", "install" (for sync) |

### Phase 4: Report

Produce a structured report with three sections:

#### 1. Integration summary

```text
Repo: [URL]
Tech stack: [detected]
Sync method: [chosen]
Assets overlaid: [N agents, N skills, N instructions]
BaseCoat version: [tag]
```

#### 2. Audit findings

Group by severity:

| # | Severity | Dimension | Finding | Recommended fix |
|---|---|---|---|---|
| 1 | 🔴 Critical | Instruction | Global instruction file is 4.8KB — adds ~4,800 tokens to every session | Split into scoped instructions by file pattern |
| 2 | 🟠 High | Agent | No agent matches the detected Python stack | Add `agents/code-review.agent.md` or a custom Python agent |
| 3 | 🟡 Medium | CI | No version drift detection workflow | Add `check-basecoat-version-callable.yml` call to CI |
| … | | | | |

If there are no findings in a dimension, state "✅ No issues found."

#### 3. Recommended next steps

List the top 3 actions by impact, in order:

1. **[Highest impact]** — what, why, exactly how
2. **[Second]** — what, why, exactly how
3. **[Third]** — what, why, exactly how

Then offer: "Would you like me to implement any of these now?"

## Tips

- If the repo URL is a GitHub URL, use the `githubRepo` tool to inspect it without cloning.
- If running inside the target repo, use the `codebase` tool to read files.
- If the user's org has a private BaseCoat fork, ask for the fork URL before generating sync commands.
- Keep the audit report scannable — use tables and emoji severity markers. Avoid walls of prose.
- Never modify the target repo without asking first. The prompt is read-only; changes happen only with explicit user approval.
